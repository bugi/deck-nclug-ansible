        ansible is

        a fictional machine capable of instantaneous or superluminal communication (Rocannon's World, Ursula K. Le Guin, 1966)

        • Well documented.
        • a tool for automating
          • deployment of software and systems
          • configuration of software and systems
          • orchestration of deployment and configuration
        • systems, as in it will talk to
          • aws
          • docker
          • libvirt
          • rackspace
          • and several others
        4000.about/0200.ansible.is

        ansible is not


        Ansible addresses the same problem space as
        • puppet
        • salt
        • chef
        • rex
        • cfengine
        4000.about/0300.ansible.is.not

        how ansible is different


        • new on the scene: 2012
          • minimal infrastructure: ssh and python
            • push / agentless
              • most others are pull
              • ansible can also pull, but push is the norm
            • yaml
              • desired states are specified in yaml
              • yaml is easy to read and write
              • salt also uses yaml, but not as well IMO
            • python
              • base is python
              • modules just have to talk JSON
            4000.about/0400.how.ansible.is.different

            orchestration


            • scenario:
              • 5003 hosts
              • deploy to {{X}} at a time
              • fail the deployment if {{Y}}% fail
            • for each host, {{X}} hosts in parallel:
              • remove service from load balancer
              • instantiate new, clean VM or container
              • pull The Build from jenkins, git, etc
              • install
              • configure
              • sanity test
              • add service to load balancer

              idempotence

              (not unique to ansible)

              (but important)



              • Modules are idempotent.
              • They will seek to avoid changing the system unless a change needs to be made.
              • They can safely be run more than once.

              (The shell module and friends are not idempotent.)


              An example...

              playbook

              % cat site.yml
---

- hosts: localhost
  tasks:

    - name: /tmp/j deserves this, honest
      file: path=/tmp/j state=absent

    - name: identify!
      debug: msg="{{ inventory_hostname }}"

              % ansible-playbook -i hosts site.yml
